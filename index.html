01. IoT 전동 킥보드 2인 탑승 감지 시스템
▼ 자세히 보기
1. 제작 동기 및 문제 인식

학교 등하굣길에서 전동 킥보드 하나에 두 명이 위태롭게 매달려 가는 모습을 자주 목격했습니다. 법적으로 금지되어 있지만, 현장 단속이 어렵다는 점이 가장 큰 문제였습니다. 단순히 "하지 마세요"라는 경고문이나 캠페인보다는, 기계가 스스로 인원을 감지하고 즉각적인 피드백(경고음)을 주는 시스템이 있다면 사고를 실질적으로 줄일 수 있을 것이라 판단하여 팀 'KICK'의 기술 담당으로 프로젝트에 참여했습니다.

2. 기술적 구현 원리 (Sensor Fusion)
Arduino Uno C++ FSR 압력 센서 초음파 센서

단일 센서만으로는 가방을 올려두거나 주행 중 진동을 사람으로 오인하는 '오탐(False Positive)' 문제가 발생했습니다. 이를 해결하기 위해 서로 다른 두 종류의 센서 데이터를 결합하는 센서 융합(Sensor Fusion) 알고리즘을 설계했습니다.

[감지 알고리즘 로직]
1. 조건 A (압력): 발판에 배치된 4개의 FSR 센서 중 3개 이상에서 압력이 감지됨.
2. 조건 B (밀집도): 전/후방 초음파 센서 거리의 합이 설정값 이하로 줄어듦 (사람 간 밀착).
3. 오차 보정 (Filtering): 주행 중 덜컹거림으로 인한 센서 튀김을 방지하기 위해, 위 두 조건이 '3회 연속(Count Logic)' 감지될 때만 최종적으로 Buzzer를 울림.

3. 나의 역할 및 문제 해결 (Troubleshooting)

[H/W: 물리적 불안정성 극복]
기술부로서 회로 설계와 센서 배치를 전담했습니다. 이론상 완벽했던 회로도 실제 킥보드 진동 환경에서는 점퍼선이 헐거워지거나 전원 공급이 불안정해지는 문제가 있었습니다. 이를 해결하기 위해 배선을 다시 정리하고, 납땜을 통해 물리적 연결을 견고하게 만들어 하드웨어 신뢰성을 확보했습니다.

[S/W: 논리적 사고의 확장]
단순히 센서 값을 읽는 코드를 넘어, '어떤 상황이 진짜 2인 탑승인가?'를 정의하는 논리 구조를 짰습니다. FSR과 초음파 센서의 교집합(AND 연산) 조건을 설정하고, 반복 검증 루프를 추가하여 시스템의 정확도를 높였습니다. 이 과정에서 전자공학과 프로그래밍이 별개가 아니라 유기적으로 연결되어 있음을 깊이 체감했습니다.

4. 아쉬운 점 및 향후 계획

초기 기획했던 '모터 속도 강제 제어' 기능까지는 구현하지 못해, 경고음 송출에 그친 점이 아쉬움으로 남습니다. 하지만 하드웨어의 한계를 소프트웨어적 필터링으로 보완하는 엔지니어링 역량을 길렀습니다. 향후에는 센서 데이터의 노이즈를 더 정밀하게 제거하기 위해 칼만 필터(Kalman Filter) 등을 학습하여 적용해보고 싶습니다.

02. DDC 창의탐구: P2P 블록체인 메신저 개발
▼ 자세히 보기
1. 탐구 배경: 단일 실패 지점(SPOF)의 위험

카카오톡 데이터센터 화재 사건을 보며 '중앙 집중형 서버'의 취약점을 깊이 체감했습니다. 편리함을 위해 모든 데이터를 한곳에 모으는 구조는 화재나 재난 시 모든 소통이 단절되는 위험을 안고 있습니다. 이에 중앙 서버가 파괴되더라도 개인 간 연결이 유지되는 '서버 없는(Serverless) P2P 메신저'를 기획하게 되었습니다.

2. 핵심 기술: Socket 통신과 블록체인
Python Socket Programming SHA-256

[탈중앙화 통신 구현]
거대 서버를 거치지 않기 위해 Python의 socket 라이브러리를 활용했습니다. 각 컴퓨터가 서버이자 클라이언트가 되는 P2P 구조를 설계하여 IP 주소만으로 직접 데이터를 주고받는 환경을 구축했습니다.

[데이터 무결성 보장]
중앙 관리자가 없으면 대화 내용의 위변조 위험이 생깁니다. 이를 막기 위해 대화 메시지를 Block 객체로 만들고, 이전 메시지의 해시값(SHA-256)을 현재 메시지에 포함시키는 체인(Chain) 구조를 적용했습니다.
