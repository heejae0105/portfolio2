💡 왜 만들었는가? (Motivation)

학교 주변에서 친구들이 킥보드를 둘이서 타다가 위험한 상황을 겪는 걸 자주 봤습니다. 법으로 금지되어 있지만, 실제 현장에서는 단속이 어렵다는 점이 문제였습니다. 그래서 '사람이 감시하지 않아도 기계가 알아서 2인 탑승을 막을 순 없을까?'라는 생각으로 프로젝트를 시작했습니다.

🛠️ 기술적 해결 과정 (Logic)
Arduino Uno C++ FSR(압력) 센서 초음파 센서

처음엔 간단할 줄 알았지만, 센서 하나만 쓰니까 오작동이 너무 심했습니다. 가방을 올려도 사람으로 인식하거나, 주행 중 덜컹거림을 탑승으로 착각하기도 했습니다. 그래서 두 가지 센서를 섞어서 쓰는 센서 융합(Sensor Fusion) 방식을 고안했습니다.

  • ✅ 조건 1: 발판 압력 센서 4개 중 3개 이상이 눌림 (발이 많다는 증거)
  • ✅ 조건 2: 앞뒤 초음파 센서 거리가 가까워짐 (사람끼리 밀착했다는 증거)

특히 주행 중에 센서 값이 튀는 현상을 잡기 위해, 이 두 조건이 '3번 연속'으로 감지될 때만 경고음이 울리도록 코드를 수정했습니다. 이렇게 하니 실수로 밟은 건 무시하고 진짜 2인 탑승만 잡아낼 수 있었습니다.

02. DDC 창의탐구: P2P 블록체인 메신저 개발
▼ 자세히 보기
1. 탐구 배경: 단일 실패 지점(SPOF)의 위험

카카오톡 데이터센터 화재 사건을 보며 '중앙 집중형 서버'의 취약점을 깊이 체감했습니다. 편리함을 위해 모든 데이터를 한곳에 모으는 구조는 화재나 재난 시 모든 소통이 단절되는 '단일 실패 지점(Single Point of Failure)' 문제를 안고 있습니다. 이에 중앙 서버가 파괴되더라도 개인 간 연결이 유지되는 통신 시스템의 필요성을 느껴, '서버 없는(Serverless) P2P 메신저'를 기획하게 되었습니다.

2. 핵심 기술: Socket 통신과 블록체인
Python Socket Programming SHA-256 P2P Network

이 프로젝트의 핵심은 두 가지 기술적 챌린지를 해결하는 것이었습니다.

[A. 탈중앙화 통신 구현]
거대 서버를 거치지 않기 위해 Python의 socket 라이브러리를 활용했습니다. Client-Server 구조가 아닌, 각 컴퓨터가 서버이자 클라이언트가 되는 P2P 구조를 설계하여 IP 주소만으로 직접 데이터를 주고받는 환경을 구축했습니다.

[B. 데이터 무결성 보장 (블록체인)]
중앙 관리자가 없으면 대화 내용의 위변조 위험이 생깁니다. 이를 막기 위해 대화 메시지를 Block 객체로 만들고, 이전 메시지의 해시값(SHA-256)을 현재 메시지에 포함시키는 체인(Chain) 구조를 적용했습니다. 누군가 과거 대화를 수정하면 이후 모든 해시값이 깨지도록 설계하여 수학적으로 신뢰를 보장했습니다.

3. 트러블 슈팅 및 결론

[네트워크 장벽 해결]
로컬호스트(127.0.0.1)에서는 잘 작동하던 코드가 실제 다른 컴퓨터와 연결할 때는 방화벽과 포트(Port) 문제로 접속이 거부되는 현상이 발생했습니다. 이를 해결하기 위해 핫스팟으로 동일 네트워크 대역을 구성하고, 방화벽 인바운드 규칙을 수정하며 네트워크 계층(OSI 7 Layer)에 대한 이해도를 높였습니다.

[최종 산출물]
인터넷망이 끊긴 상황을 가정하고 노트북 2대를 연결하여 채팅을 시연하는 데 성공했습니다. 기업의 서버가 멈춰도 개인의 소통은 멈추지 않아야 한다는 '디지털 정보 주권'의 가치를 기술적으로 증명해낸 의미 있는 탐구였습니다.
